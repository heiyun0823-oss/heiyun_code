## 5.1 常见疑问解答

### Q1：为什么文件里到处是 `.js` 后缀的 import，这不是 TS 项目吗？

**示例**：`packages/ai/src/openai.ts` 第 1 行

```typescript
import type { GenerateChunk, GenerateRequest, ... } from "./types.js";
//                                                         ↑↑ .js 不是 .ts
```

**解答**：这是 TypeScript 在 ESM 模式下的"约定"。TS 源码里写 `./types.js`，编译时 TS 不会改这个后缀（因为编译后确实是 `.js` 文件）。Node.js 在运行时根据这个路径找到 `dist/types.js`。

简单说：**写代码时写 `.js`，TS 编译器知道你要引的是 `.ts` 源文件，编译后路径恰好正确**。

---

### Q2：`as import("@heiyun/ai").ToolParameter` 是什么语法？

**示例**：`packages/tools/src/read.ts` 第 18 行

```typescript
} as import("@heiyun/ai").ToolParameter;
```

**解答**：这是一种**内联类型标注**，完整叫法是"类型断言 + 动态类型导入"。

分解理解：
```typescript
// 普通 import type（写在文件顶部）
import type { ToolParameter } from "@heiyun/ai";
const def: ToolParameter = { type: "object", ... };

// 等价的内联写法（不需要顶部 import）
const def = { type: "object", ... } as import("@heiyun/ai").ToolParameter;
```

为什么这样写？因为这个文件中另一个函数（`resolveSafePath`）不需要 `ToolParameter` 类型。如果顶部 `import type`，TS 会在编译时引入这个依赖。用内联写法可以让依赖关系更精确。

**对 JS 程序员来说**：把它理解为"告诉 TS 这个对象应该长成 ToolParameter 的样子"即可。

---

### Q3：`for await...of` 和普通 `for...of` 有什么区别？

**示例**：`packages/agent-core/src/loop.ts` 第 100 行

```typescript
for await (const chunk of provider.generateStream(req)) { ... }
```

**解答**：

| | `for...of` | `for await...of` |
|---|---|---|
| 遍历对象 | 同步可迭代对象（数组、Set 等） | **异步**可迭代对象（AsyncGenerator） |
| 每次迭代 | 立刻拿到值 | `await` 等待值就绪 |
| 使用场景 | `[1, 2, 3]` | 网络流、文件流、定时器 |

在这个项目里，`generateStream()` 返回的是 `AsyncGenerator<GenerateChunk>`——LLM 的回复是一个异步流，每个 chunk 是网络传过来的，需要等。

用 JS 的话说：**如果 `for...of` 是吃已经端上桌的菜，`for await...of` 就是在厨师做一道端一道的餐厅里吃**。

### Q4：`Partial<T>` 是什么意思？

**示例**：`packages/agent-core/src/loop.ts` 第 21 行

```typescript
function mergeToolCallDelta(
  toolCalls: ToolCall[],
  delta: Partial<ToolCallDelta>  // ← Partial
): void { ... }
```

**解答**：`Partial<T>` 是 TS 内置的工具类型，意思是"把 T 的所有属性变成可选的"。

```typescript
interface ToolCallDelta {
  index: number;
  id?: string;
  type: "function";
  function?: { name?: string; arguments?: string; };
}

// Partial<ToolCallDelta> 等价于：
{
  index?: number;       // 变成可选
  id?: string;          // 本来就可选
  type?: "function";    // 变成可选
  function?: { ... };   // 本来就可选
}
```

为什么这里用 `Partial`？因为流式传输中的 delta 片段**可能只包含部分字段**（比如只传了 `function.arguments` 而没有 `id`）。

### Q5：React 是怎么用在终端里的？没浏览器怎么渲染？

**解答**：这个项目使用的是 **ink** 库。ink 把 React 的"虚拟 DOM → 真实 DOM"替换为"虚拟 DOM → 终端字符串"。

| | 浏览器 React | ink（终端 React） |
|---|---|---|
| 组件写法 | `<div>`、`<span>` | `<Box>`、`<Text>` |
| 样式 | CSS | 颜色字符串、`borderStyle` |
| 渲染目标 | DOM 树 | 终端字符输出 |
| 事件处理 | `onClick` | `useInput`（键盘） |

原理：React 本身是一个**与渲染目标无关**的 UI 框架。react-dom 把组件渲染到 DOM，ink 把组件渲染到 ANSI 转义序列（终端能理解的彩色字符）。

**对 JS 程序员来说**：如果你会写 React 组件（`function MyComp() { return <div>hello</div> }`），把 `div` 换成 `Box`、`span` 换成 `Text`，就能写终端 UI 了。

### Q6：为什么要用 monorepo？不能把所有代码放一个包里吗？

**解答**：技术上完全可以放一个包。但 monorepo 的好处是：

1. **边界清晰**：`@heiyun/ai` 不能引用 `@heiyun/tools`（依赖反向会报错），强制架构保持线性
2. **独立测试**：可以单独跑 `cd packages/ai && npm test`，不依赖其他包
3. **独立发布**：如果以后想把 `@heiyun/ai` 单独发到 npm 给别人用，直接 `npm publish -w packages/ai`
4. **编译缓存**：不修改的包不用重新编译

在这个项目里，包之间的依赖是严格线性的（ai ← tools ← agent-core ← cli），所以 monorepo 的优势特别明显。

### Q7：为什么工具调用参数用 JSON 字符串而不是 JS 对象？

**示例**：LLM 返回的工具调用参数是 `'{"path":"package.json"}'`（字符串），不是 `{ path: "package.json" }`（对象）。

**解答**：这是 OpenAI Function Calling 协议规定的格式。LLM 生成的参数必须是 JSON 字符串，因为：

1. LLM 输出的是文本，不能直接输出 JS 对象
2. JSON 是语言无关的标准格式
3. API 规范如此（所有兼容 OpenAI 的 API 都遵循这个约定）

所以 `toolRegistry.execute()` 里永远有一行：

```typescript
// tool-registry.ts 第 50 行
params = JSON.parse(toolCall.function.arguments);  // 字符串 → 对象
```

---

### Q8：项目里的报错信息为什么用中文？编程不是应该用英文吗？

**解答**：这是一个面向**中文开发者**的工具。关键考量：

- 用户看到的错误提示用中文 → 更友好（`"未知工具: xxx"` vs `"Unknown tool: xxx"`）
- 发给 LLM 的 system prompt 用英文 → 因为 LLM 训练数据主要是英文，英文指令效果更好
- 代码里的变量名、函数名用英文 → 编程惯例

这个"中英混用"策略是刻意设计的，不是随意的。

## 5.2 JS → TS 学习路线建议

你已经有了 JS 基础。以下是建议的学习路径，分三个阶段：

### 第一阶段：语法适应（1-2 天）

目标：能在 TS 中写 JS 代码不报错。

| 学习内容 | 项目中的参考 |
|---------|------------|
| 基础类型标注（`: string`、`: number`） | `packages/ai/src/types.ts` 第 20-26 行 |
| `interface` 和 `type` 的区别 | 文档 01 第 4 节 |
| 联合类型（`|`）和可选属性（`?`） | `packages/ai/src/types.ts` 第 12、23-26 行 |
| 函数参数和返回值标注 | `packages/tools/src/read.ts` 第 47-49 行 |

**练习**：打开 `packages/ai/src/types.ts`，尝试在脑海里把每个 interface 翻译成"这个对象必须有哪些字段"的口头描述。

### 第二阶段：理解设计（3-5 天）

目标：理解 TS 如何帮助构建大型项目。

| 学习内容 | 项目中的参考 |
|---------|------------|
| 泛型（`Map<K,V>`、`Promise<T>`） | `packages/agent-core/src/tool-registry.ts` 第 11 行 |
| `implements` 和面向接口编程 | `packages/ai/src/openai.ts` 第 3 行 |
| 异步生成器（`async *`） | `packages/ai/src/openai.ts` 第 50-152 行 |
| 工具类型（`Omit<>`、`Partial<>`） | `packages/agent-core/src/session.ts` 第 15 行 |
| 模块化设计和依赖管理 | 文档 00 第 4 节（依赖关系图） |

**练习**：跟着文档 04 的数据流图，用纸笔画出"一条用户输入怎么走完整个系统"，标注每个环节的文件和行号。

### 第三阶段：项目实战（持续）

目标：能独立修改和扩展这个项目。

| 学习内容 | 可做的事 |
|---------|---------|
| 添加一个新工具 | 在 `packages/tools/` 下新建 `mytool.ts`，仿照 `read.ts` 的模式 |
| 修改 system prompt | 编辑 `packages/agent-core/src/system-prompt.ts` |
| 添加一个新的斜杠命令 | 仿照 `packages/cli/src/slash-commands/login.tsx` |
| 写单元测试 | 参考 `packages/ai/src/openai.test.ts` |

## 5.5 给你的最后建议

1. **不要试图一次记住所有东西**。这 6 份文档是"地图"，不是"课本"。遇到具体问题时回来查，比通读一遍更有效。

2. **最好的学习方法是用起来**。打开终端，输入 `heiyun`，跟它对话，让它帮你改代码。观察每一步它在做什么、每个工具结果长什么样。

3. **TS 的类型系统是渐进式的**。你不需要写"完美"的类型。哪怕只是加个 `: string` 的注解，也比纯 JS 安全得多。从简单的标注开始，慢慢加更复杂的类型。

4. **看这个项目的设计文档**。`docs/superpowers/specs/` 目录下有很多设计文档，记录了每个功能"为什么这样设计"的决策过程。理解了设计意图，理解代码就会容易很多。
